# 車載音声認識の全体像

## この章で学ぶこと

- 音声入力からTTSまでの処理フロー
- 文字化、意味理解、状態管理、車両実行の違い
- 不具合をコンポーネント単位で切り分ける方法

## なぜ必要なのか

車載音声機能は複数のコンポーネントが結果を受け渡すシステムです。全体像を知らないと、「認識できなかった」という現象だけを見て、マイク、ASR、NLU、車両APIのどこに原因があるか判断できません。

## 全体像

```text
Mic → Wake Word → ASR → NLU → Dialog Manager → Domain Service → Vehicle API
                                                              ↓
User ← Speaker ← TTS ← Response ------------------------------+
```

各要求には同じ関連IDを付け、入力音声から車両APIの結果まで追跡できるようにします。

## 基本用語

- **ASR**: 音声を文字列へ変換する。
- **NLU**: 文字列をIntentやSlotへ変換する。
- **Dialog Manager**: 会話状態から確認、追加質問、実行などを決める。
- **Domain Service**: Navigation、Phone、Media、Climateなどの機能判断を担う。
- **Vehicle API**: 車両機能へ要求し、成功・失敗・現在値を返す境界。

## 詳細解説

発話「エアコンを22度にして」では、ASRが文字列を返し、NLUが`SetTemperature`と`22°C`を抽出します。Dialog Managerは情報が十分か、確認が必要かを判断します。Climateサービスは値の範囲や走行状態を確認し、Vehicle APIへ要求します。成功後にTTSで結果を通知します。

重要なのは、各段階の成功が次段階の成功を保証しない点です。ASR結果が正しくてもNLUが誤分類する場合があり、NLU結果が正しくても車両側が要求を拒否する場合があります。

## 車載での利用例

- ナビ: 目的地名を認識し、候補選択後に経路案内を開始する。
- 電話: 連絡先を検索し、同名候補を確認して発信する。
- 空調: 設定値を正規化し、許容範囲を確認して温度を変更する。
- メディア: 再生対象を検索し、Audio Focusを取得して再生する。

## 設計時の注意点

- 起動、認識、確認、実行、終了、キャンセルをStateとして定義する。
- 各コンポーネントのTimeoutと、Timeout後のFallbackを決める。
- 通話、警告音、ナビ案内、外部アシスタントとの競合を定義する。
- 走行中制限や権限確認は、音声解釈ではなく実行直前にも検証する。

## レビュー観点

- **機能**: 正常系と確認・キャンセル・再試行がつながっているか。
- **品質**: 精度とLatencyがコンポーネント別に測定できるか。
- **ログ**: 関連ID、時刻、入出力、State、失敗理由が残るか。
- **安全性・運用**: 誤操作の抑止、通信断やプロセス停止からの復旧があるか。

## よくある不具合

- ASRの部分結果を最終結果として実行してしまう。
- 古いSessionのNLU結果が新しい要求に適用される。
- Vehicle APIが失敗したのにTTSが成功と案内する。
- Timeout後もマイクやAudio Focusを解放しない。

## 確認問題

1. ASR結果が正しいのに別の機能が実行された場合、どこを確認しますか。
2. Vehicle APIがTimeoutした場合、ユーザー通知と内部状態をどう扱いますか。
3. 一つの要求を端から端まで追跡するログ項目を挙げてください。

## まとめ

車載音声機能は、音声入力、文字化、意味理解、対話判断、車両実行、応答の連鎖です。レビューでは各コンポーネントの責務と境界、Timeout、競合、関連ログを分けて確認します。
