# LLM連携

## この章で学ぶこと

- LLM、Tool Calling、RAG、Agentの役割
- 従来型NLUとの使い分け
- 車両操作へつなぐ際の安全境界

## なぜ必要なのか

LLMは多様な表現や質問へ柔軟に対応できますが、出力が常に正確・一定とは限りません。自然な応答を生成する処理と、実際の車両操作を許可する決定的な処理を分離する必要があります。

## 全体像

```text
発話・Context → LLM → Tool候補（構造化引数）
                         ↓
             Schema検証 → Policy・権限・車両状態確認 → Vehicle API
                         ↓ NG
                    確認・拒否・従来型Fallback
```

## 基本用語

- **Tool Calling**: モデルが呼び出す機能名と引数を構造化して提案する仕組み。
- **RAG**: 検索した資料を入力に加えて回答を生成する構成。
- **Agent**: 複数の判断やTool利用を組み合わせる処理主体。
- **Grounding**: 回答を取得した情報や観測結果に結びつけること。
- **Guardrail**: 入出力や実行を制約・検証する仕組み。

## 詳細解説

従来型NLUはIntentとSlotが限定された操作で、予測可能性、Latency、オフライン動作を設計しやすい方式です。LLMは曖昧な質問、言い換え、マニュアル検索に強みがあります。すべてを置き換えるのではなく、機能のリスクと必要な柔軟性で使い分けます。

LLMが返したTool名と引数は「実行命令」ではなく候補です。許可リスト、Schema、値域、権限、走行状態、確認要否を通常コードで検証してからVehicle APIへ渡します。

## 車載での利用例

「曇りを取って」と言われた場合、LLMはデフロスター操作を候補にできます。ただし、対象車種の機能、現在状態、許可された引数を検証し、曖昧なら確認します。車載マニュアルへの質問はRAGで該当箇所を検索し、根拠を取得できない場合は断定しません。

## 設計時の注意点

- 読み取り、可逆操作、不可逆・安全関連操作で権限と確認を分ける。
- Prompt Injectionを想定し、取得文書やユーザー入力に実行権限を与えない。
- 通信断、Rate limit、モデルTimeout時の従来型NLUや定型応答を用意する。
- 音声、位置、連絡先、車両状態の送信範囲と保持方針を最小化する。

## レビュー観点

- **機能**: LLMの担当範囲と従来型Fallbackが明確か。
- **品質**: タスク成功率、Tool引数精度、Latency、根拠取得率を測るか。
- **ログ**: モデル・Prompt版、Tool候補、検証結果、実行結果を追えるか。
- **安全性・運用**: Policy、権限、Privacy、停止スイッチ、版切り戻しがあるか。

## よくある不具合

- LLMが生成したTool引数を検証せず、そのまま車両APIへ渡す。
- 取得資料にない内容を、マニュアル記載事項として回答する。
- モデルTimeout時にSessionとAudio Focusを解放しない。
- Promptやモデル更新後の挙動変化を版管理できない。

## 確認問題

1. Tool Calling結果を実行する前に必要な検証を挙げてください。
2. 従来型NLUを残す方がよい車載機能の例と理由を書いてください。
3. RAGが根拠を取得できない場合、どう応答しますか。

## まとめ

LLMは柔軟な理解と生成に使えますが、車両操作の権限を直接持たせません。構造化、決定的検証、確認、Fallback、Privacy、版管理を安全境界として設計します。
