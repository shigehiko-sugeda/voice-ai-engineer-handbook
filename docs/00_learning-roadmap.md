# 学習ロードマップ

## この章で学ぶこと

- 車載音声システムを学ぶ順序
- 各章と演習の対応
- 「読んだ」から「設計をレビューできる」へ進むための到達基準

## なぜ必要なのか

車載音声システムは、ASRだけでは成立しません。マイク入力、意味理解、対話管理、音声出力、車両機能との連携がつながって動きます。さらに、SiriやGoogle Assistant、通話、ナビ案内などと音声資源を取り合う場面もあります。

個別技術だけを先に学ぶと、不具合が起きたときに「どのコンポーネントの問題か」「どのログが必要か」を切り分けにくくなります。そのため、まず処理全体を把握し、次に各技術を学び、最後に競合制御・評価・アーキテクチャを横断してレビューします。

## 全体像

学習は次の4段階で進めます。

1. **処理の流れをつかむ**
   [車載音声認識の全体像](01_overview.md)で、音声入力から車両操作、TTS応答までの流れを確認します。
2. **音声対話の構成要素を理解する**
   [音声・オーディオ基礎](02_audio-basics.md)から[TTS](07_tts.md)までを読み、Wake Word、ASR、NLU、Dialog Managerの責務を区別します。
3. **車載固有の制約を理解する**
   [車両連携](08_vehicle-integration.md)、[Audio FocusとArbitration](09_audio-focus-and-arbitration.md)、[認証観点](10_certification.md)で、車両状態や他の音声機能との競合を扱います。
4. **測定し、壊れにくい構成をレビューする**
   [評価指標](11_metrics.md)、[アーキテクチャ](12_architecture.md)、[LLM連携](13_llm-integration.md)で、品質評価、障害復旧、安全な機能拡張を検討します。

## 基本用語

- **Pipeline**: マイク入力から応答・車両操作までをつなぐ処理の流れ。
- **State**: 待機中、認識中、確認中、実行中など、システムの現在の状態。
- **Arbitration**: 複数の機能がマイクやスピーカーを要求したとき、どれを優先するか決めること。
- **Fallback**: 認識失敗、通信断、タイムアウトなどの際に、安全に終了または代替処理へ移ること。
- **レビュー観点**: 設計の不足やリスクを見つけるための確認軸。この教材では、機能、品質、ログ、認証、安全性、運用に分けます。

## 学び方

各章は、次の順序で使います。

1. 「この章で学ぶこと」と「全体像」を読み、対象コンポーネントの入出力を確認する。
2. 自分の言葉で「誰から何を受け取り、誰へ何を渡すか」を説明する。
3. 確認問題に答える。分からない場合は本文へ戻る。
4. 対応する演習で、正常系だけでなく、タイムアウト、競合、通信断、キャンセルも検討する。
5. 判断に必要なログを挙げる。結果だけでなく、状態遷移、要求元、時刻、失敗理由を追えるか確認する。

AI家庭教師を使う場合は、いきなり模範解答を求めず、「ヒントを1つ」「私の回答の不足点だけ」「次に確認すべきログは何か」のように依頼すると、自分で設計判断する練習になります。

## 車載での利用例

発話「エアコンを22度にして」を例にすると、次の章がつながります。

1. マイクが車室内の音声を取得する（02）。
2. Wake WordまたはVoice Buttonで受付を開始する（03、10）。
3. ASRが音声を文字列へ変換する（04）。
4. NLUがClimateドメイン、温度設定Intent、22度というSlotを抽出する（05）。
5. Dialog Managerが確認の要否やTimeoutを判断する（06）。
6. Vehicle APIを通じて空調へ要求する（08）。
7. TTSが結果を通知する（07）。
8. 通話やナビ案内と重なった場合は、Audio FocusとArbitrationが優先順位を決める（09）。

この流れのどこかで失敗したとき、各コンポーネントの状態と受け渡したデータをログで追えることが、実務レビューの最低条件です。

## 設計レビューの到達基準

学習後は、設計資料に対して少なくとも次の質問ができる状態を目指します。

- **機能**: 起動から終了までの正常系と、キャンセル・再試行の流れが定義されているか。
- **品質**: 認識精度とLatencyの目標値、測定条件、合否判定があるか。
- **ログ**: コンポーネントをまたいで要求を追跡でき、状態遷移や失敗理由を特定できるか。
- **認証**: Voice Button、Wake Word、Mic Routing、Audio Focusなどの確認事項が整理されているか。
- **安全性**: 走行中の操作制限、誤作動、低Confidence時の確認、危険な操作の抑止が考慮されているか。
- **運用**: 通信断、プロセス停止、設定不整合からの検知・復旧・切り戻しが定義されているか。

## よくある学習上のつまずき

- ASRとNLUをまとめて「音声認識」と捉え、文字化の失敗と意味解釈の失敗を区別できない。
- 正常系だけを追い、Timeout、キャンセル、割り込み後の復帰を見落とす。
- 最終結果だけをログに残し、どのStateで止まったか特定できない。
- 外部アシスタントとの競合を個別機能の問題として扱い、マイクとAudio Focusの所有権を確認しない。
- 認証要件を推測で断定し、公開仕様と関係者への確認事項を分けない。

## 確認問題

まず各問について、対応する章を1つ選んでから答えてください。

1. ASR結果が正しいのに、意図しない車両機能が実行されました。どのコンポーネントとログを確認しますか。
2. ナビ案内中に外部アシスタントが起動しました。音声の競合をレビューする項目を4つ挙げてください。
3. 音声要求が途中で止まる不具合について、コンポーネントをまたいで原因を追うために必要なログ項目は何ですか。

## まとめ

最初にPipeline全体をつかみ、次に各コンポーネントの責務を学び、車載固有の競合と制約へ進みます。最後に評価指標とアーキテクチャを使って、正常系だけでなく、Timeout、競合、障害復旧、ログまでレビューできる状態を目指します。
