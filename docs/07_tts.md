# TTS

## この章で学ぶこと

- テキストを聞き取りやすい音声へ変換するTTSの役割
- 車載向けPromptと読み上げ設計
- 割り込み、Queue、Audio Focusの扱い

## なぜ必要なのか

運転中は画面を注視できないため、TTSは主要なフィードバック手段です。内容が長い、結果と発話が矛盾する、警告音やナビ案内と重なる、といった問題は操作性と安全性に直結します。

## 全体像

```text
Dialog結果 → Prompt生成 → TTS合成 → 再生Queue → Audio Focus → Speaker
```

## 基本用語

- **Prompt**: ユーザーへ伝える応答文。
- **SSML**: 読み方、間、強調などを指定する記法。
- **Barge-in**: 再生中の発話によってTTSを止め、入力へ切り替えること。
- **Ducking**: 背景音の音量を下げてTTSを聞きやすくすること。
- **Playback Queue**: 複数の再生要求を順序づける待ち行列。

## 詳細解説

Promptは短く、結果と次の行動が分かる形にします。例えば成功確認は「22度に設定しました」、失敗時は「車両から応答がありません。もう一度お試しください」のように、実行結果に基づいて生成します。

合成完了と再生完了は別のEventです。再生前にAudio Focusを取得し、終了・キャンセル・エラー時に解放します。古いSessionのPromptはQueueから除去します。

## 車載での利用例

ナビ案内中に空調設定の結果を返す場合、案内の重要度とタイミングに応じて待機、ダッキング、中断を選びます。警告音など安全優先の音をTTSが遮らないようにします。

## 設計時の注意点

- 成功、失敗、確認、再質問のPromptを実行結果と対応づける。
- 数字、住所、人名、単位の読み方と多言語Fallbackを定義する。
- Queueの優先度、重複排除、期限、キャンセルを設計する。
- Barge-in後のTTS停止完了とASR開始の順序を定義する。

## レビュー観点

- **機能**: 表示とTTSが同じ結果を伝えるか。
- **品質**: 合成Latency、再生開始Latency、聞き取りやすさを測るか。
- **ログ**: Prompt ID、Session ID、合成・再生時刻、停止理由を残すか。
- **安全性**: 警告音や通話より不適切に優先されないか。

## よくある不具合

- Vehicle API失敗後に「設定しました」と読み上げる。
- キャンセル済みSessionのTTSが遅れて再生される。
- 人名や住所の読みが誤り、別候補と区別できない。
- 再生終了後もAudio Focusを保持して音楽が復帰しない。

## 確認問題

1. TTS合成完了と再生完了を分ける理由は何ですか。
2. Barge-in時に必要な処理順序を説明してください。
3. 古いPromptの再生を防ぐログと識別子を挙げてください。

## まとめ

TTSは合成だけでなく、Prompt、Queue、Audio Focus、割り込みを含みます。実行結果との整合性と、終了時の確実な資源解放を確認します。
