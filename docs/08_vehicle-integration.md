# 車両連携

## この章で学ぶこと

- Navigation、Phone、Media、Climate、Vehicle Settingsとの接続
- Vehicle APIと車載ネットワークの境界
- 実行可否、非同期応答、失敗を扱う方法

## なぜ必要なのか

NLUが正しいIntentを返しても、車両がその操作を受け付けるとは限りません。車両状態、権限、走行中制限、通信状態によって結果が変わるため、音声側は要求送信と実行完了を区別する必要があります。

## 全体像

```text
Dialog Manager → Domain Service → Vehicle API → Gateway / ECU
                      ↑                 ↓
               制約・権限確認 ← 結果・現在値・失敗理由
```

## 基本用語

- **Domain Service**: 機能領域ごとの判断とAPI変換を担う層。
- **Vehicle API**: 上位機能と車両制御を分離するインターフェース。
- **ECU**: 車両機能を制御する電子制御ユニット。
- **Request ID**: 非同期要求と応答を対応づける識別子。
- **Idempotency**: 同じ要求が重複しても結果を壊さない性質。

## 詳細解説

音声側がCANやECUを直接操作するとは限りませんが、APIの入力範囲、単位、権限、Timeout、結果コードを理解する必要があります。「要求を送れた」と「車両状態が変わった」は別です。必要に応じて完了応答または状態更新を確認してから成功を通知します。

## 車載での利用例

「窓を開けて」という要求では、対象窓、開度、車速、チャイルドロックなどの条件が関係します。許可されない場合は実行せず、一般化した理由をユーザーへ返します。安全上の制約を音声解釈だけで回避しません。

## 設計時の注意点

- APIの型、単位、範囲、Error code、Timeoutを契約として定義する。
- 送信、受理、実行完了、状態反映を区別する。
- Retry可能な失敗と、再試行してはいけない失敗を分ける。
- Session終了後の遅延応答と重複要求を安全に処理する。

## レビュー観点

- **機能**: Domainごとに実行条件とユーザー通知が定義されているか。
- **品質**: API応答時間と状態反映時間を別々に測るか。
- **ログ**: Session ID、Request ID、要求値、結果コード、現在値を残すか。
- **安全性・運用**: 走行中制限、権限、通信断、ECU停止を扱えるか。

## よくある不具合

- API送信成功だけで操作成功のTTSを返す。
- 摂氏と華氏、割合と絶対値などの単位を取り違える。
- TimeoutをRetryして同じ操作を二重実行する。
- 車両状態更新の遅延を失敗と判断し、表示と実状態がずれる。

## 確認問題

1. 要求送信成功と実行完了を区別する方法を説明してください。
2. Retry前に確認すべきAPI特性は何ですか。
3. 車両連携不具合を追跡するログ項目を挙げてください。

## まとめ

車両連携では、意味理解の結果を安全なAPI要求へ変換し、実行結果を確認して応答します。制約、非同期性、重複、Timeoutを境界仕様としてレビューします。
