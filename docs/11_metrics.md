# 評価指標

## この章で学ぶこと

- WER、Intent Accuracy、Slot F1などの精度指標
- Latency、False Wake Rate、Miss Wake Rateの測定
- 再現可能な評価条件と実車評価

## なぜ必要なのか

「認識精度が良い」「応答が速い」だけでは合否を判断できません。対象データ、分母、測定区間、車両条件を揃えて初めて、版間比較と不具合の切り分けができます。

## 全体像

```text
品質目標 → データ・条件 → 実行 → 指標集計 → 条件別分析 → 合否・改善
```

## 基本用語

- **WER**: `(置換 + 削除 + 挿入) / 正解単語数`で表すASR誤り率。
- **Intent Accuracy**: Intentが正解した発話の割合。
- **Slot Precision / Recall / F1**: Slot抽出の正確さと網羅性。
- **Latency**: 定義した起点から終点までの所要時間。
- **Percentile**: p50、p95など、遅いケースを含む分布上の位置。

## 詳細解説

平均値だけでは一部の極端に遅い応答を見落とします。LatencyはWake検出、ASR最終化、NLU、Vehicle API、TTS開始などに分解し、p50とp95などを確認します。

False Wake RateとMiss Wake Rateは分母と観測時間を明記します。実車では、停車・市街地・高速、座席、窓、空調、音楽、話者、通信状態をメタデータとして残し、条件別に比較します。

## 車載での利用例

「高速走行時にナビ目的地の認識率が低下する」という課題では、同じ発話セットを停車時と高速時で比較します。音声入力品質、WER、Intent、Slot、Endpoint、Latencyを段階別に見ることで、音響とNLUのどちらが支配的かを判断します。

## 設計時の注意点

- 指標ごとに対象、正解データ、分母、除外条件を定義する。
- 学習・調整データと最終評価データを分離する。
- 全体値だけでなくDomain、条件、話者などの層別結果を確認する。
- 個人情報を含む音声・発話ログの同意、匿名化、保存期間を決める。

## レビュー観点

- **機能・品質**: ユーザー成功率と各コンポーネント指標が対応しているか。
- **ログ**: 同一Sessionを音声入力から車両結果まで追えるか。
- **安全性**: 危険操作の誤受理を全体平均に埋めていないか。
- **運用**: モデル、設定、車両、データ版を固定して再現できるか。

## よくある不具合

- 平均Latencyだけを見てp95の悪化を見落とす。
- False Wakeの分母が走行時間と発話回数で混在する。
- 正解ラベルの誤りをモデル誤りとして集計する。
- 評価セットへ最適化し、実車の未知条件で性能が落ちる。

## 確認問題

1. WERが改善してもタスク成功率が下がる例を考えてください。
2. End-to-end Latencyを分解する時刻を挙げてください。
3. 実車評価結果に必ず付ける条件情報は何ですか。

## まとめ

評価指標は数値だけでなく、分母、測定区間、条件、版情報とセットです。End-to-end結果と段階別指標を関連づけ、平均に隠れた失敗を確認します。

## 参考資料

- [SCTK, the NIST Scoring Toolkit — NIST](https://github.com/usnistgov/SCTK): ASRのAlignment、Word Error、複数Systemの統計比較を行う公開Toolkitです。
- [Media Capture and Streams — W3C](https://www.w3.org/TR/mediacapture-streams/): Sampling Rate、Latency、Channel Countなど、音声評価条件に関係する標準化された属性を確認できます。
