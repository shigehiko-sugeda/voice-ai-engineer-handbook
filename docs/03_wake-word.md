# Wake Word

## この章で学ぶこと

- Wake Wordによる音声受付の開始
- False WakeとMiss Wakeの違い
- 車載環境での起動・抑制・終了設計

## なぜ必要なのか

Wake Wordは常時待ち受けから対話Sessionへ切り替える入口です。誤起動が多いと音楽や会話を妨げ、起動漏れが多いと機能を使えません。起動精度だけでなく、起動後のマイク所有権や終了処理まで設計対象です。

## 全体像

```text
Idle → Wake候補検出 → しきい値判定 → Mic取得 → ASR Listening
  ↑                         ↓ 競合・禁止状態
  +---------- 抑制または終了 ----------+
```

## 基本用語

- **False Wake**: Wake Wordではない音を起動語と誤判定すること。
- **Miss Wake**: 実際のWake Wordを検出できないこと。
- **Threshold**: 起動判定に使うスコアのしきい値。
- **Pre-roll**: 検出直前の音声を保持し、後続発話の先頭欠けを防ぐ仕組み。
- **Suppression**: 通話中など、条件に応じて検出や起動を抑えること。

## 詳細解説

しきい値を下げると起動しやすくなる一方、False Wakeも増えやすくなります。そのため、利用者、車速、ノイズ、再生コンテンツなどを揃えた評価が必要です。検出後はASRの準備完了を待ち、Pre-rollを含む音声を渡します。

## 車載での利用例

音楽再生中に運転者がWake Wordに続けて「自宅へ案内して」と発話します。システムは再生音による誤起動を抑えつつ、起動後はAudio Focusとマイクを取得し、「自宅へ」の先頭を欠かさずASRへ渡します。

## 設計時の注意点

- 通話、バック走行、外部アシスタント起動中などの抑制条件を定義する。
- Wake WordとVoice Buttonが同時に起きた場合の重複Sessionを防ぐ。
- 起動音、Listening表示、プライバシー表示のタイミングを揃える。
- ASR開始失敗や無発話Timeout時にマイクを必ず解放する。

## レビュー観点

- **機能**: Idleから終了までの状態遷移と再起動条件があるか。
- **品質**: False Wake、Miss Wake、起動Latencyを条件別に測るか。
- **ログ**: 検出スコア、しきい値、抑制理由、マイク取得結果を残すか。
- **安全性・運用**: 禁止状態と、誤起動を調査する再現手段があるか。

## よくある不具合

- ラジオ番組内の類似語で繰り返し起動する。
- 検出からASR開始までの間に発話先頭が欠ける。
- 外部アシスタント起動中に自社Wake WordもSessionを開始する。
- Timeout後もListening表示やマイク所有状態が残る。

## 確認問題

1. False Wakeを減らすとMiss Wakeが増え得る理由を説明してください。
2. Wake検出後にASRへ渡す情報を挙げてください。
3. 通話中のWake Word動作は、どの状態とログで確認しますか。

## まとめ

Wake Wordは検出モデルだけでなく、Session開始、競合抑制、表示、マイク解放まで含む機能です。精度と起動後の状態遷移を一緒にレビューします。
