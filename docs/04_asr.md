# ASR

## この章で学ぶこと

- 音声をテキストへ変換するASRの役割
- Streaming ASR、Endpoint Detection、Confidence
- 車載ノイズ下での評価と切り分け

## なぜ必要なのか

NLU以降はASRの文字列を入力にするため、文字化の誤りは後段へ伝播します。一方、意図の誤解釈や車両実行失敗までASRの問題と扱うと原因を見誤ります。ASRの入出力と確定タイミングを明確にします。

## 全体像

```text
音声Frame → 音響処理 → Streaming ASR → 部分結果 → Endpoint → 最終結果
```

## 基本用語

- **Partial Result**: 発話途中に更新される未確定の認識結果。
- **Final Result**: 発話終端の判定後に確定した結果。
- **Endpoint Detection**: 発話終了を判断する処理。
- **Confidence**: 認識結果の信頼度を表す値。値の意味は実装ごとに確認する。
- **WER**: 置換、削除、挿入を基に単語列の誤りを測る指標。

## 詳細解説

Streaming ASRは音声を受け取りながら部分結果を返し、体感Latencyを短縮します。ただし、部分結果は後から変化するため、不可逆な車両操作の実行には最終結果を使います。

Endpointを短くすると応答は速くなりますが、考えながら話すユーザーの発話を途中で切る可能性があります。長くすると取りこぼしは減る一方、応答開始が遅れます。車載ではロードノイズ、空調音、音楽、同乗者、座席位置ごとに評価します。

## 車載での利用例

「東京駅を経由して横浜へ案内して」という発話では、途中の間を発話終了と誤判定すると目的地情報が欠けます。最終結果、単語時刻、Endpoint理由を残し、NLUへどの文字列を渡したか確認します。

## 設計時の注意点

- 部分結果と最終結果の利用範囲を分ける。
- 無発話、最大発話長、通信、サーバー応答のTimeoutを区別する。
- 低Confidence時の再発話、候補提示、確認をドメイン別に決める。
- オンラインASR利用時は通信断時のFallbackと音声データの扱いを定義する。

## レビュー観点

- **機能**: 開始、部分結果、確定、キャンセルのイベント順序が定義されているか。
- **品質**: WERだけでなくEndpoint誤りとLatencyを条件別に測るか。
- **ログ**: Session ID、音声区間、部分・最終結果、Confidence、終了理由を残すか。
- **運用**: モデル版、言語、設定値を再現できるか。

## よくある不具合

- 部分結果で車両操作を開始し、最終結果との差分を戻せない。
- 無音区間が短く設定され、発話を途中で確定する。
- 同乗者の声を運転者の指示として認識する。
- ASR ConfidenceとNLU Confidenceを混同して調査する。

## 確認問題

1. 部分結果を車両操作に直接使うリスクは何ですか。
2. 「認識が遅い」を切り分ける時刻を3つ挙げてください。
3. 高速走行時だけ誤りが増える場合、ASR以外に何を確認しますか。

## まとめ

ASRは音声を文字列にし、最終結果をNLUへ渡します。精度、Endpoint、Latencyを分け、音声条件と設定を再現できるログを残します。
